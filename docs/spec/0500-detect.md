# 사양 0500 — 형태 감지, 증거, 발견사항

대상 모듈: `src/detect.rs`(0500), `src/evidence.rs`(0600), `src/findings.rs`(0700)
관련 요구사항: 0501–0506, 신규 0507, 0601–0606, 0701–0706, 1103, 1104

## 1. 감지 규칙표 (이 표가 전부다 — 규칙 추가·해석·생략 금지)

매치는 inventory `entries`의 상대 경로 기준이다. "이름"은 경로의 마지막
구성 요소를 뜻한다. 별도 표기가 없으면 대소문자를 구분한다.

| 규칙 | 매치 조건 | category | priority | 내용 열람 | 발견사항 묶음 |
| --- | --- | --- | --- | --- | --- |
| D01 | 이름이 `package.json` (모든 깊이) | manifest | 정보 | O — scripts 파싱 | 규칙당 1건 |
| D02 | D01 파일의 `scripts`에 `preinstall`, `install`, `postinstall`, `prepare` 중 하나 이상 존재 | auto-exec-hook | 중간 | O — 키가 있는 줄 번호 | 파일·키당 1건 |
| D03 | 이름이 `package-lock.json`, `pnpm-lock.yaml`, `yarn.lock` | manifest | 정보 | X | 규칙당 1건 |
| D04 | 이름이 `Cargo.toml`, `Cargo.lock` | manifest | 정보 | X | 규칙당 1건 |
| D05 | 이름이 `build.rs` (모든 깊이) | auto-exec-hook | 중간 | X | 파일당 1건 |
| D06 | 이름이 `Makefile`, `makefile`, `GNUmakefile` | build-automation | 낮음 | X | 규칙당 1건 |
| D07 | 이름이 `Dockerfile`로 시작, 또는 `docker-compose.yml`, `docker-compose.yaml`, `compose.yml`, `compose.yaml` | container | 낮음 | X | 규칙당 1건 |
| D08 | 경로가 `.github/workflows/`로 시작하고 ext가 `yml`/`yaml` | ci-automation | 낮음 | X | 규칙당 1건 |
| D09 | ext가 `sh`, `bash`, `zsh` (0504) | shell-script | 중간 | X | 규칙당 1건 |
| D10 | 이름이 `.envrc` | auto-exec-hook | 높음 | X | 파일당 1건 |
| D11 | 경로가 `.vscode/tasks.json`으로 끝남 | auto-exec-hook | 중간 | X | 파일당 1건 |
| D12 | `kind=dir`이고 이름이 `.husky` | auto-exec-hook | 중간 | X | 디렉터리당 1건 |
| D13 | 이름이 `.env`, 이름이 `.env.`으로 시작, 이름이 `id_rsa`/`id_dsa`/`id_ecdsa`/`id_ed25519`로 시작, ext가 `pem`/`p12`/`pfx`, 이름에 `credential` 또는 `secret` 포함(대소문자 무시) | secret-candidate | 중간 | **기본 금지** | 파일당 1건 |

- "규칙당 1건": 매치된 모든 경로를 증거로 모아 발견사항 하나를 만든다.
- "파일당 1건"(또는 키당): 매치 한 건마다 발견사항 하나를 만든다.
  자동 실행 지점(D02, D05, D10, D11, D12)과 비밀값 후보(D13)는 개별
  경로가 곧 행동 항목이므로 묶지 않는다.
- 한 파일이 여러 규칙에 걸리면 걸린 규칙 전부 적용한다
  (예: `.env.sh` → D09와 D13).
- D13에 걸린 파일은 다른 규칙이 내용 열람을 요구해도 열지 않는다.
  D13이 항상 이긴다(1103, 1104).
- D13은 제외 규칙이 아니다. 기본 검사에서 원문을 열지 않는 대신,
  설치·빌드·테스트·실행·모델 입력 전에 별도 결정을 요구하는 미열람
  보류 항목으로 보고한다(1107–1109, 1113–1115, 1120–1122,
  1210–1213).
- 사용자가 해당 파일에 민감정보가 없다고 판단하거나, 미상의 출처에서
  받은 스크립트로 의심된다고 판단한 경우에만 별도 민감 후보 진단에서
  2중 승인과 경로별 승인으로 내용을 분석할 수 있다. 1차 승인은 민감
  후보 진단 시작 승인이고, 2차 승인은 원문 분석에 포함할 경로 목록
  확정이다. 2차 승인 없는 원문 분석은 금지한다. 이 별도 진단은 최소
  기능판 기본 검사 범위 밖이다(1409, 1410).

## 2. 내용 열람 규칙 (coverage와 연결)

- 내용을 여는 파일은 D01 매치 파일뿐이다(D02는 D01이 연 내용을 재사용).
- 단, D13에 함께 걸린 파일은 D01 매치 여부와 관계없이 기본 검사에서
  열지 않는다.
- 열기 전 첫 8KiB에 NUL 바이트가 있으면 바이너리로 보고 내용 분석을
  포기하며, coverage의 `skip_reasons.binary`에 센다.
- 연 파일은 coverage `read_files`에 `{"path","bytes"}`로 기록하고
  `bytes_read_total`에 합산한다(0905).
- `package.json` JSON 파싱 실패 시: D02는 건너뛰고, findings의
  `limitations`에 고정 문장 `package.json을 파싱하지 못했다: <경로>`를
  추가한다. 읽기는 했으므로 read_files에는 남는다.
- D02 줄 번호: 파일 원문에서 `"<키이름>"` 문자열이 처음 나오는 줄
  (1부터 셈). 발견 못 하면 lines는 null.

## 3. 증거 — evidence.rs (0600)

- `EvidenceStore::add(path, kind, lines, summary, excerpt) -> EvidenceId`.
- id는 `E0001`부터 추가 순서대로. 같은 실행 안에서 재사용·결번 금지(0601).
- `kind` 허용값: `file-presence` | `content-line` | `symlink-record` |
  `secret-name`.
- `lines`는 `content-line`일 때만 채운다(9002 확정). 나머지는 null.
- `excerpt`는 최대 200자(문자 기준, 초과분 절단). `secret-name`이면
  항상 null(0604).
- 규칙별 증거 kind와 summary 고정 문구:

| 규칙 | kind | 증거 summary (고정) |
| --- | --- | --- |
| D01 | file-presence | `자바스크립트 패키지 매니페스트가 존재함` |
| D02 | content-line | `package.json에 {키} 스크립트가 정의됨` |
| D03 | file-presence | `의존성 잠금 파일이 존재함` |
| D04 | file-presence | `러스트 매니페스트 파일이 존재함` |
| D05 | file-presence | `카고 빌드 스크립트가 존재함` |
| D06 | file-presence | `빌드 자동화 파일이 존재함` |
| D07 | file-presence | `컨테이너 정의 파일이 존재함` |
| D08 | file-presence | `지속 통합 워크플로 파일이 존재함` |
| D09 | file-presence | `셸 스크립트가 존재함` |
| D10 | file-presence | `.envrc 파일이 존재함` |
| D11 | file-presence | `편집기 자동 태스크 정의 파일이 존재함` |
| D12 | file-presence | `깃 훅 배포 디렉터리가 존재함` |
| D13 | secret-name | `비밀값으로 보이는 이름의 파일이 존재함 (내용 미열람)` |

- D02의 excerpt는 해당 줄 원문(앞뒤 공백 제거, 200자 절단).
- 심볼릭 링크는 감지 규칙이 아니지만, inventory의 skipped 기록이 그대로
  한계 문장으로 이어진다. 증거로 만들 필요는 없다.

## 4. 발견사항 — findings.rs (0700)

- `Finding::new(id, category, priority, summary, detail, limitation, evidence)`
  는 `evidence`가 비어 있으면 `Err`를 반환한다. 증거 없는 발견사항은
  타입 수준에서 만들 수 없다(0605, 0606, 1304).
- id는 `F0001`부터, 규칙 번호 순(D01→D13), 같은 규칙 안에서는 경로
  사전순으로 부여한다. 결정적이어야 한다(T10).
- `category` 허용값: `auto-exec-hook` | `build-automation` | `container` |
  `ci-automation` | `shell-script` | `secret-candidate` | `manifest`.
- `priority` 허용값: `정보` | `낮음` | `중간` | `높음` (9001 확정, 0702).
- 어떤 문구에도 `악성`, `안전` 단어를 넣지 않는다(0506, 0704).

### 공통 한계 목록 (findings.json `limitations` — 항상 들어가는 고정 문장)

```text
무시 규칙(.gitignore 등)은 적용하지 않고 전 항목을 나열했다.
심볼릭 링크는 따라가지 않았다.
형태 감지는 이름 기반이며 파일 내용 분석은 package.json의 scripts에 한정되었다.
비밀값 후보 파일은 기본 검사에서 원문을 열지 않았으며, 별도 승인 전에는
내용 판단에서 제외되었다.
```

여기에 상황별 문장(깃 상태 계산 실패, package.json 파싱 실패,
낮은 확신 — 사양 0900의 V06)을 덧붙인다.
