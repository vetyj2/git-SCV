# 사양 0200 — 검사 흐름, 원본 식별, 안전 경계

대상 모듈: `src/inspect.rs`(0200), `src/source.rs`(0300), `src/safety.rs`(1100)
관련 요구사항: 0201–0205, 0301–0305, 0901, 0961–0984, 1101–1106

## 1. 단계 순서 (0201 — 고정, 변경 금지)

| 순서 | 단계 이름(run.json `stages[].name` 값) | 하는 일 |
| --- | --- | --- |
| 1 | `source` | 원본 식별 (이 문서 3절) |
| 2 | `walk` | 파일 탐색 (사양 0400) |
| 3 | `detect` | 형태 감지 (사양 0500) |
| 4 | `evidence` | 증거 기록 (사양 0500) |
| 5 | `findings` | 발견사항 생성 (사양 0500) |
| 6 | `sensitive` | 민감 후보 승인 모드와 후보별 처리 기록 |
| 7 | `validate` | 메모리 내 정합성 검증 (사양 0900의 V02–V04, V06–V24) |
| 8 | `artifacts` | `run.json`을 제외한 inspect 산출물 22종 기록 (사양 0900) |
| 9 | `report` | 전체 inspect 산출물 23종 존재와 report.md 무실행 문구 최종 확인 (V01, V05) |

- `inspect::run`은 위 순서로만 호출을 조립한다.
- 입력 검증(사양 0100의 4절)은 단계 이전에 끝나 있어야 하며, 단계 시작
  전에 `--out` 디렉터리를 만든다(`create_dir_all`).

## 2. 실패 정책 (0202, 0203)

- 각 단계는 성공 시 `{"name":"…","status":"ok","error":null}`,
  실패 시 `{"name":"…","status":"failed","error":"<사유 문자열>"}`을 남긴다.
- 한 단계가 실패하면 뒤 단계는 `"status":"skipped"`로 기록하고 실행하지
  않는다. 단, **run.json은 어떤 경우에도 마지막에 기록한다**(0202, 0203).
  run.json 기록 자체가 실패하면 stderr 메시지만 남기고 코드 3으로 종료한다.
- 단계 실패 시 그 시점까지 만들어진 데이터(예: source 정보)는 산출물로
  남긴다. 즉 artifacts 단계 이전에 실패해도 run.json과 (있다면)
  source.json은 기록을 시도한다.
- 검사 한도는 `coverage.json.local_limits`와 inventory/snapshot 메타데이터에
  기록한다. 한도 초과는 insufficient-coverage reason code로 이어진다.

## 3. 원본 식별 — source.rs (0300)

출력 구조는 사양 0900의 source.json 스키마를 따른다. 채우는 규칙:

- `input.raw`: 사용자가 입력한 문자열 그대로. `input.kind`: 항상
  `"local-path"`(최소 기능판, 0021).
- `resolved_path`: `std::fs::canonicalize` 결과(0301).
- **깃 판정**: `<root>/.git`이 존재할 때만 깃 저장소로 취급한다.
  상위 디렉터리로 올라가며 찾는 discover 방식은 금지한다 — 검사 대상은
  입력 디렉터리 그 자체다. 따라서 `gix::open(<root>)`을 쓰고, 실패하면
  `"git": null`로 두고 검사를 계속한다(0302, 0303).
- 깃 저장소일 때 채울 값:
  - `branch`: HEAD가 가리키는 가지 짧은 이름. 분리 HEAD면 null,
    `detached: true`.
  - `commit`: HEAD 커밋의 40자리 16진수. 커밋이 하나도 없는 저장소면 null.
  - `dirty`: gix 상태 계산으로 미커밋 변경(작업 트리 또는 인덱스) 존재
    여부. 상태 계산이 오류로 실패하면 `dirty: null`로 두고
    findings.json의 `limitations`에 아래 고정 문장을 추가한다:
    `깃 변경 상태를 계산하지 못해 미커밋 변경 여부를 확인하지 않았다.`
  - `remotes`: 깃 설정의 원격 이름과 주소를 이름 사전순으로 기록만 한다.
    네트워크 접근(fetch, ls-remote 등)은 절대 하지 않는다(0304).
- `snapshot`: 로컬 `inspect` 실행에서는 null을 기록한다. `snapshot` 명령이
  다운로드, 체크섬 검증, 안전 압축 해제를 통과해 기존 검사 흐름을 호출한
  경우에는 스냅샷 URL의 query/fragment를 제거한 값, 소문자 SHA-256,
  압축 형식, 추출 경로를 기록한다(1402L).
- 기본 `inspect`는 원격 저장소 주소를 가져오지 않는다. 이후 원격
  스냅샷은 별도 명령에서만 다루며, 1차 정책은 압축 내려받기와 사용자
  제공 체크섬 검증이다(9005). 격리 복제는 git 메타데이터가 꼭 필요한
  별도 승인 확장으로 둔다.

### 왜 gix인가 (구현자가 바꿀 수 없는 결정)

`git` 외부 명령 호출은 대상 저장소의 `core.fsmonitor` 설정으로 임의
명령이 실행될 수 있다 — 이 도구가 잡으려는 바로 그 공격이다. 깃 정보는
반드시 `gix` 크레이트로 in-process 수집한다. 외부 프로세스 생성 0회는
시험 T09가 기계적으로 강제한다.

## 4. 안전 경계 — safety.rs (1100)

- `pub fn assert_inside(out_root: &Path, target: &Path) -> Result<(), ScvError>`:
  `target`의 존재하는 가장 가까운 부모를 canonicalize한 결과가
  canonicalize된 `out_root`로 시작하지 않으면 `Inspect` 오류를 낸다.
  artifacts 단계는 **모든 파일 쓰기 직전에** 이 함수를 호출한다(1105, 1106).
- 대상 저장소 내부의 어떤 파일도 실행하지 않고, 어떤 외부 프로세스도
  생성하지 않는다(1101). 크레이트 전체에서 프로세스 생성 API 사용 금지
  (시험 T09의 금지 문자열: `Command::`, `.spawn(`, `.output(`, `.status(`).
- 비밀값 후보 파일(감지 규칙 D13)은 내용을 열지 않는다(1103, 1104).
  이 규칙은 detect 단계 구현에 있고, safety.rs는 경로 봉쇄만 맡는다.
- 대상 저장소에는 어떤 쓰기도 하지 않는다. 대상 저장소는 읽기 전용으로만
  연다.

## 5. 시각과 run_id

- 모든 시각은 UTC, RFC3339 초 단위(`2026-06-12T12:00:00Z`)로 기록한다.
- `run_id` = `scv-` + 시작 시각을 `YYYYMMDDTHHMMSSZ`로 압축한 문자열.
  예: `scv-20260612T120003Z`. 모든 산출물의 `run_id`는 같은 값이다.
